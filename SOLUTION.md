# Solution Notes

## What was built

REST API backend for a loyalty-points wallet. Go 1.26, stdlib `net/http`, GORM + PostgreSQL.

```
cmd/server/        — server package: router, middleware stack, TLS, admin seed
internal/config/   — Config struct, YAML + env merge, DSN()
internal/db/       — GORM open, AuditPlugin, TransactionPlugin
internal/handlers/ — one Handler struct, one file per domain
internal/middleware/ — recovery → logging → auth (with skip-path list)
internal/requestid/ — X-Request-Id generation
internal/models/   — generated by relspec (DO NOT EDIT)
openapi.json       — OpenAPI 3.1 spec
```

## Middleware stack

```
Recovery → RequestID → Logging → Auth → mux
```

Auth receives a skip-path list at registration (`/healthz`, `/version`, `/`, `/api/v1/auth/login`, openapi, swagger). All other routes require a valid non-expired bearer token.

## Audit plugin

Registered as a GORM plugin. Fires `After("gorm:create/update/delete")` on `public.user`, `public.account`, `public.account_transaction`. Each write produces one `audit_event` row and one `audit_detail` row per changed column, within the same DB transaction. Reflection is used to extract column names and values from `Statement.Dest`.

## Transaction plugin

Registered as a GORM plugin. Fires `Before("gorm:create")` on `public.account_transaction` only.

| Check | Sentinel error | HTTP |
|---|---|---|
| `(reference, rid_account)` already exists | `ErrDuplicateTransaction` | 409 |
| `spend` would drive balance below zero | `ErrInsufficientBalance` | 400 |

The spend check issues `SELECT ... FOR UPDATE` on the account row to prevent a TOCTOU race between concurrent spend requests. Both errors are checked in the handler with `errors.Is` before falling through to the generic unique-violation check.

## Schema fix

The original DBML had `id_user`, `id_account`, `id_audit_event` as plain `bigint` (no `increment`). This was an oversight — those IDs are DB-generated. Updated all three to `bigserial [pk, not null, increment]` and re-ran `make generate`.

## Balance recalculation

After every transaction insert or delete, `recalcBalance` recomputes the account balance from scratch:

```sql
SELECT COALESCE(SUM(CASE WHEN action='earn' THEN amount ELSE -amount END), 0)
FROM public.account_transaction WHERE rid_account = ?
```

This runs in the same GORM transaction as the insert/delete, so the balance column is always consistent.

## Admin seeding

On startup, if no `role = 'admin'` row exists and `ADMIN_PASSWORD` is set, one admin user is created. Safe to skip if the DB already has an admin.

## CSV batch ingestion (`POST /api/v1/transactions/batch/csv`)

Accepts `Content-Type: text/csv`. Header row: `ref,account_id,kind,points,occurred_at`.

Processing is two-pass:

**Pass 1 (before any DB writes):** parse every row, validate fields, check that each `account_id` exists and the caller has ownership. Any failure here returns immediately with a row number in the error message. No DB state is changed.

**Pass 2 (single atomic transaction):** all `tx.Create` + `recalcBalance` calls run inside one `gorm.DB.Transaction(...)`. If any row fails (duplicate reference, insufficient balance, DB error), the entire transaction rolls back and the error response includes the failing row number.

`occurred_at` is accepted as RFC3339, `2006-01-02T15:04:05`, `2006-01-02 15:04:05`, or `2006-01-02`. Empty value defaults to `NOW()`.

## AI workflow

### What I asked and what came back

**Session start — spec reading**
Prompted the AI to read `AGENTS.md`, `go-skill.md`, and all existing files before writing anything. This produced a clear implementation plan in `todo.md` and a set of clarifying questions before any code was written.

**Clarifying questions (two rounds)**

*Round 1 — before coding started:*
- "Are auth login/logout endpoints part of the implementation?" — Yes, implement both. Accepted as-is; the spec described them but omitted them from the endpoint table.
- "Swagger UI: CDN or embedded?" — CDN. Accepted; simpler and no binary bloat.

*Round 2 — discovered during model inspection:*
- "Is `id_user`/`id_account` as plain `bigint` (no autoincrement) intentional?" — No, oversight. The AI caught this by reading the generated model tags and realising GORM would INSERT zero-value PKs. DBML updated and models regenerated before any handler code was written.

**Implementation — largely accepted without edits**
The AI implemented all phases (config → db → middleware → handlers → server → openapi) in order. One generated snippet was broken (`ListTransactions` had a dead variable dereference from an incomplete refactor) and was immediately self-corrected on the next edit. No other logic required correction.

**Transaction plugin**
Asked for two GORM before-create guards. The AI chose:
- A separate `TransactionPlugin` (not merged into `AuditPlugin`) for single-responsibility.
- `Session(&gorm.Session{NewDB: true})` for sub-queries inside the callback to avoid contaminating the in-flight statement.
- `clause.Locking{Strength: "UPDATE"}` on the account row for the balance check to prevent concurrent-spend races.

Accepted without changes. The pattern is idiomatic and the locking rationale was sound.

**Admin rules clarification**
After flagging that admin-only rules should cover delete-user, delete-account, and access to other members' accounts, a grep confirmed all guards were already in place from the initial implementation. No changes needed.

**CSV batch ingestion**
Asked for a CSV endpoint. One clarifying question was asked: what does `account_id: member-123` (a string in the example) map to in the integer schema? Answer: it's the integer `id_account`; the example was illustrative. No schema change needed.

Initial implementation called `captureTransaction` per row (each with its own transaction). A follow-up requirement — "batch must be in a single SQL/GORM transaction" — prompted a refactor:
- Extracted `buildTransactionRecord` and `mapTxError` helpers to eliminate duplication.
- Split the CSV handler into two passes: field-validation + ownership checks first (outside any transaction, with row-number context), then all DB writes in one outer `h.db.Transaction(...)`.

The two-pass design was chosen over a full refactor of `captureTransaction` to minimise risk to the existing single/JSON-batch paths. Accepted.
